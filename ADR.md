# ADR-001: レストランレビュー管理システムの構築

## ステータス

採用（Accepted）― 2026-04

---

## コンテキスト（背景と課題）

- Google Map や食べログにレビュー投稿をしているが数が多くなってきた
- 外部サービスに依存せず、マスタ情報は手元で持っておきたい
- 将来別のサービスを利用する場合に、活用しやすい形で管理しておきたい

これらの課題を解決するため、個人利用のレストランレビュー管理システムを自作する。

---

## 決定内容

### 技術スタック

| 領域 | 採用技術 | 理由 |
| --- | --- | --- |
| フロントエンド | TypeScript + Tailwind CSS | 型安全な UI 開発 |
| バックエンド | Python + Django + REST API | 汎用的な API サーバー |
| データベース | PostgreSQL | 正規化されたリレーショナルデータの管理 |
| ORM | SQLAlchemy | DB 操作の抽象化とマイグレーション管理 |

### 設計方針

- DB は正規化する。プルダウンの選択肢はマスタテーブル（`masters`）で管理し、画面から動的に読み込む
- 削除はソフトデリート（`deleted_at` カラム）とし、データは物理削除しない
- 要件変更時は既存データを移行できるよう、マイグレーション SQL を都度作成する
- API ベース URL など環境依存の値は環境変数経由で注入し、ソースにハードコードしない
- DB 値の取得失敗時はエラーハンドリングを行い、ユーザーにメッセージを表示する
- API エラー発生時はバックエンドのログに HTTP メソッド・パスを記録する。バリデーションエラーはフィールド名・エラー内容をログとレスポンス（`details`）に含め、フロントエンドでも HTTP ステータスと対象 URL を画面表示する
- 背景画像は `public/bg.svg` を `index.css` の `body` に `background-image` で設定し、`cover` でフルスクリーンに表示する
- 管理者モードの認証状態は `sessionStorage` で保持し、ブラウザタブを閉じると解除される
- パスワード検証はサーバーサイドの `POST /api/admin/verify` エンドポイントで行い、フロントエンドにパスワードを保持しない

#### 写真管理

- 1 件につき複数枚の写真を登録できる。ファイル選択・ドラッグ&ドロップいずれも対応し、複数ファイルを一括追加できる
- 写真は 90 度単位で回転でき、状態を保存できる
- 写真の並べ替えができる。◀ ▶ ボタンによる操作に加え、ドラッグ&ドロップでも並べ替えられる
- 1 枚目の写真がサムネイルとして使われる
- オリジナル写真・サムネイルともに DB に BYTEA 型で保存する
- 写真は `/api/photos/{id}`、サムネイルは `/api/photos/{id}/thumb` エンドポイントで配信する

#### 一覧画面

- サムネイル付きで一覧表示できる
- 店名・最寄り駅・訪問年月でキーワード検索できる
- ジャンル・利用シーンで絞り込みできる
- 並び順を指定できる
- 50 件ごとにページングする
- 表示形式を大アイコン・中アイコン・小アイコン・グリッドから選択できる
- 他画面から戻った際に検索条件・表示形式を保持する

#### 編集・削除

- 情報を 1 件単位で更新できる
- 写真サムネイルをクリックすると原寸表示できる
- 削除はコンファーム確認後、論理削除とする

#### ダウンロード機能

- 一覧表示中のレビューデータを CSV または JSON 形式でダウンロードできる（写真は対象外）

#### 管理者モード

- 初期状態は閲覧者モードとなっている
- 一覧画面から管理者モードに切り替えられる
- 管理者モードではレビューの登録・編集・削除操作が可能となる
- 管理者モードに入るにはパスワードが必要となる
- 管理者パスワードは `.env` の `ADMIN_PASSWORD` で定義する

---

## 結果・影響

視認しやすい形でレストランレビューの管理ができる

### ポジティブ

- レビュー情報を一元管理でき、外部サービスへの依存から脱却できる
- 写真・評価・コメントを記録することで、再訪や紹介の際の判断材料になる
- マスタテーブルによる選択肢管理でデータの統一性が保たれる

### トレードオフ・制約

- 個人運用前提のためマルチユーザー対応は持たない。管理者モードは誤操作防止を目的としたシンプルなパスワード保護であり、本格的な認証基盤ではない
- 写真・サムネイルをすべて DB に保存するため、サーバー移行時にファイル移動が不要になる
- 写真データが大きい場合は DB サイズが増大する点に注意が必要

## コードレビュー観点（2026-04）

本プロジェクトのコードレビューで継続的に確認する観点。

| # | カテゴリ | 観点 |
| --- | --- | --- |
| 1 | セキュリティ | ファイル名パラメータを受け取るエンドポイントでは `os.path.basename()` でサニタイズし、元の値と一致しない場合は 400 を返す（パストラバーサル対策） |
| 2 | セキュリティ | ファイルアップロード時は拡張子チェックではなく Pillow の `img.verify()` で実バイナリを検証する。拡張子偽装（.exe を .jpg にリネーム等）を防ぐため |
| 3 | 正確性 | DB コミット失敗時は `try/except` で `rollback` を行い、中途半端な状態でデータが残留しないようにする |
| 4 | 保守性 | 同一のフィルタ・クエリ処理が複数箇所に重複する場合はヘルパー関数に抽出する |
| 5 | パフォーマンス | ほぼ全クエリで使用するカラムにはインデックスを付与する。追加時はマイグレーション SQL も作成する |
| 6 | 保守性 | 環境依存の値（API ベース URL 等）は環境変数経由で読み込む定数ファイルに切り出し、ソースにハードコードしない |
| 7 | UX | データ取得系の非同期処理は `catch` ブロックを設け、失敗時にユーザーへエラーメッセージを表示する |
| 8 | 保守性 | どこからも参照されないコンポーネント・モジュールは削除する |
| 9 | 保守性 | linter を適用して保守性を維持する |
| 10 | 保守性 | formatter を適用して保守性を維持する |
| 11 | ベストプラクティス | `Array.map` で生成する React 要素の `key` には配列インデックスではなく、変化しない一意な識別子を使用する |
| 12 | セキュリティ | `.env` は `.gitignore` で除外し、リポジトリにコミットしない。テンプレートとして `.env.example`（ダミー値）を管理する |
| 13 | セキュリティ | CORS の `allow_methods` / `allow_headers` はワイルドカード `"*"` を使わず、使用するメソッド・ヘッダーを明示的に列挙する |
| 14 | セキュリティ | レスポンスに `X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY`、`Referrer-Policy`、`Content-Security-Policy` ヘッダーを付与する |
| 15 | セキュリティ | 例外メッセージをそのまま HTTP レスポンスに返さない。内部エラーは固定の日本語メッセージに置き換え、詳細はサーバーログに留める |
| 16 | セキュリティ | Pydantic の入力スキーマには `max_length`・`ge`/`le` 等のフィールド制約を付与し、過大な入力値を境界で弾く |
| 17 | バグ予防 | すべての `<button>` 要素には必ず `type` 属性を明示する。省略するとデフォルトの `type="submit"` として扱われ、フォーム内外を問わず意図しないフォーム送信が発生する。フォーム送信ボタンは `type="submit"`、それ以外はすべて `type="button"` を付与する |
| 18 | UX | ポップアップ・ドロップダウン等の浮動要素は `ReactDOM.createPortal` で `document.body` 直下に描画し、親要素の `overflow` や stacking context による表示崩れを防ぐ |
| 19 | 可読性 | 同じ変換式が3回以上出現する場合は名前付きヘルパー関数に抽出する。繰り返しが消えることで各行の意図が一目で伝わる（例: `float(v) if v is not None else None` → `_maybe_float(v)`） |
| 20 | 可読性 | 変数名は値の実態を正確に表す。`count()` の戻り値を格納しているのに `max_order` と命名するなど、処理内容と名前が乖離しないようにする |
| 21 | 可読性 | 同じ式を2回以上呼び出す場合は変数に格納して再利用する（例: `model_dump(exclude_unset=True)` の二重呼び出し）。パフォーマンスの無駄を防ぐとともに、ロジックの変更が一箇所で済む |
| 22 | 可読性 | 条件式に到達不能なコードを含めない。`"day" in err.lower()` と `"Day" in err` のように一方が他方を完全に包含する場合、後者はデッドコードとなり読者を混乱させる |
| 23 | 可読性 | TypeScript の関数引数は camelCase で命名し、API リクエスト送信時にスネークケースのキーへマッピングする。言語の命名規約（camelCase）と外部 API の仕様（snake_case）を関数シグネチャ上で混在させない |
| 24 | 可読性 | マジックナンバーには導出ロジックを説明するコメントを付与する（例: `length: 41` → `// 5.0 から 1.0 まで 0.1 刻み → 41 = (5.0 - 1.0) / 0.1 + 1`） |
